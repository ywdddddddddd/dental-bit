# 基于DAG二叉树网络的AI口腔预问诊系统测试研究

## ——口腔健康解密卡模拟患者问卷测试方法与结果分析

---

## 摘要

**背景：** 口腔疾病早期症状鉴别是基层医疗的痛点。基于有向无环图（DAG）二叉树网络的AI预问诊系统通过症状驱动的交互式问答，为患者提供初步的疾病风险评估。然而，此类专家系统的测试评估缺乏标准化的方法与数据集。

**目的：** 设计一种轻量级的模拟患者问卷测试架构，对覆盖72种口腔疾病的DAG预问诊引擎进行系统性测试，验证其对不同疾病类别、紧急级别和人口统计学特征的识别能力。

**方法：** 采用Node.js + jsdom构建最小化测试框架，从生产环境HTML中提取评分引擎与DAG问题树。设计30份模拟患者真实病例，覆盖牙体牙髓、牙周、黏膜、颌面关节、其他/系统五大疾病类别及绿色/黄色/红色三个紧急级别，通过自动化DAG遍历完成全链路测试。测试指标包括首诊准确率、红色警告触发率、类别覆盖率和得分分布分析。

**结果：** 30份模拟病例中，首诊准确率为90.0%（27/30），红色警告触发率为13.3%（4/30）。牙体牙髓类准确率88.9%（8/9），黏膜类准确率100%（9/9），颌面关节类准确率100%（4/4），系统类准确率50%（1/2），牙周类准确率60%（3/5）。健康对照组首诊得分12.3分，低于15分阈值，未触发误报警。

**结论：** 该DAG预问诊引擎在常见口腔疾病的症状鉴别上表现出较高的准确率，尤其在黏膜病变（含癌前病变）和颌面关节类疾病方面表现优异。牙周类疾病的鉴别粒度不足，缺乏更精细的子类分化；缺乏特定的智齿冠周炎和放射性口炎的专属DAG路径。测试架构轻量、可复现，适用于类似专家系统的快速验证。

**关键词：** DAG二叉树网络；AI预问诊；口腔疾病；模拟患者测试；专家系统评估

## 1. 引言

口腔疾病是全球最常见的非传染性疾病之一，影响约35亿人口[1]。早期鉴别诊断对改善预后至关重要，但基层医疗资源分布不均导致大量患者错过了最佳治疗时机。近年来，基于规则推理的专家系统在医学预诊领域重新受到关注，尤其是基于症状驱动的动态问诊路径设计。

"口腔健康解密卡"是一个纯前端实现的AI预问诊系统，覆盖72种口腔疾病，采用DAG（Directed Acyclic Graph）二叉树网络作为问诊路径的调度引擎。系统通过多阶段症状深挖（主诉→症状→病史→生活方式），结合累积乘性评分、互斥抑制、因果链推理等高阶逻辑，对患者症状群进行加权推断。

然而，该系统的测试面临两个独特挑战：（1）DAG路径的非线性特征使得传统的线性回归测试方法不适用；（2）72种疾病的排列组合空间巨大，难以穷尽测试。传统上，此类专家系统依赖临床专家逐一验证，成本高、效率低且不可重复。

本文提出一种轻量级的自动化模拟患者测试架构，无需修改生产代码，通过注入30份经临床文献验证的典型患者画像，自动遍历DAG二叉树，生成标准化测试报告。本文的组织结构如下：第2节概述系统架构；第3节详细阐述测试方法与技术细节；第4节分析测试结果；第5节讨论系统局限与改进方向；第6节给出结论。

## 2. 系统概述

### 2.1 整体架构

"口腔健康解密卡"是一个单一HTML文件（约1421行）的纯客户端应用，无服务器后端、无数据库依赖。其核心组件包括：

| 组件 | 功能 | 规模 |
|------|------|------|
| 疾病知识库（DISEASES） | 72种口腔疾病的元数据 | 100+条属性/疾病 |
| 疾病关系图谱（DISEASE_GRAPH） | 因果链、互斥组、共病对 | 20条因果链 + 19组互斥 + 22对共病 |
| DAG问诊二叉树（QUESTION_TREE） | 30+问题节点，多分支路由 | 6顶层入口 + 5条专科路径 + 10个全局问题 |
| 评分引擎（OralTriageEngine） | 累积乘性评分 + 互斥抑制 + 因果推理 | ~160行规则逻辑 |

### 2.2 DAG二叉树问诊路径

系统的问诊路径呈树形拓扑，但不完全线性——不同入口分支导向不同的专科路径，最终汇聚到10个统一全局问题（年龄、性别、病史、家族史、生活习惯等）。图1展示了主要的路径拓扑结构。

**入口节点（q_entry）** 提供6个顶层分支：
1. 🦷 牙疼/冷热疼/咬合疼 → 牙体牙髓专科路径
2. 🩸 牙龈出血/红肿/松动 → 牙周专科路径
3. 👄 溃疡/白斑/糜烂 → 黏膜专科路径
4. 🕳️ 黑洞/缺损/变色 → 硬组织专科路径
5. 🗣️ 关节响/脸肿 → 颌面专科路径
6. 🏜️ 口干/口臭/烧灼 → 系统因素路径

每条专科路径包含2-4个症状深度挖掘问题，之后进入10个统一的全局问题，收集人口统计学和生活方式因子，最终触发评分引擎计算。

### 2.3 评分引擎机制

评分引擎采用多阶段加权累积策略：

1. **原始得分（Delta Accumulation）：** 每次回答立即累加对应疾病的原始分值
2. **锁定机制（Lock）：** 特定答案可彻底排除某些疾病（分数锁定为0）
3. **风险乘性累积（Risk Accumulation）：** 生活方式风险因子（吸烟、饮酒、嚼槟榔等）以乘法方式累积，而非简单加法
4. **保护乘性累积（Protection Multiplier）：** 良好口腔习惯（正确刷牙、定期检查、含氟牙膏）提供小于1.0的保护性乘子
5. **年龄兼容性过滤：** 超出疾病典型年龄范围的患者得分乘以0.3
6. **性别偏倚校正：** 匹配性别的疾病得分乘以1.3，不匹配乘以0.6
7. **互斥抑制（Mutual Exclusion）：** 同一互斥组中仅保留最高分者，其余乘以0.15
8. **因果链增量（Causal Chain Bonus）：** 上游高分疾病对下游后继疾病提供20%加成
9. **共病放大（Comorbidity Amplification）：** 存在共病关系的疾病对相互提供1.2倍加成
10. **智能矫正规则：** 针对特定临床场景的硬编码矫正（如ANUG高分时抑制牙龈炎/牙周炎）

最终输出标准化得分（精确到0.1），按得分降序排列前8名。

## 3. 测试方法

### 3.1 测试架构设计

针对该系统的三个特征——纯客户端执行、DAG非线性路由、无外部依赖——我们设计了最小化测试架构（图2）。

**技术选型：**

| 组件 | 技术 | 用途 |
|------|------|------|
| 运行环境 | Node.js v22 | JavaScript运行时 |
| DOM模拟 | jsdom v26 | 加载并执行生产HTML中的IIFE代码 |
| 引擎提取 | 代码注入 | 在IIFE结束前插入`window.__Engine = OralTriageEngine`导出 |
| 测试定义 | CommonJS模块 | 30份模拟病例的结构化答案映射 |
| 报告生成 | 文件I/O | JSON（机器可读）+ Markdown（人类可读）+ TXT（摘要） |

**设计原则：**

1. **零修改生产代码：** 通过jsdom在内存中注入导出代码，不修改`index.html`源文件
2. **全链路测试：** 从`engine.init()`到`engine.finalize()`完整遍历，不跳过任何环节
3. **多选问题正确处理：** 引擎原生的`answerBranch()`方法对多选问题存在路由缺陷——调用第一个分支后自动切换`currentNode`至下一问题，导致后续分支应用到错误的问题节点。测试框架通过保存/恢复`universalQueue`状态实现正确的多选处理
4. **可复现性：** 所有测试用例以确定性答案映射定义，每次运行的结果完全一致

### 3.2 测试执行流程

```
初始化引擎 → 获取入口问题 (q_entry)
  ↓
循环: currentNode ≠ null
  ↓
查找患者答案映射 [qId → branchIndex]
  ↓
是否为多选问题？
  ├─ 是: 保存universalQueue → 迭代应用所有选中分支的delta/riskTag/lock → 恢复队列 → 前进1步
  └─ 否: 调用engine.answerBranch(idx)
  ↓
记录问答数据
  ↓
currentNode = null → engine.finalize() → engine.ranking(8)
  ↓
收集结果: { 排名, 得分, 访问路径, 红色警告, 年龄/性别门 }
```

- **首诊准确率：** 引擎排名第一的疾病与目标疾病的匹配率

### 4.4 典型个案分析

#### 成功案例：P20 口腔癌高危病例

62岁男性，吸烟饮酒30余年，舌缘溃疡超过三周不愈合，伴硬结、麻木、体重下降。患者回答路径经过黏膜专科路径（q_mucosa_can_scrape → q_mucosa_white_or_ulcer → q_mucosa_hand_foot_rash → q_mucosa_behcet_triad → q_mucosa_drug_trigger → q_mucosa_gingival_desquamation → q_mucosa_ulcer_duration → q_mucosa_cancer_redflags），加上全局问题中的吸烟/饮酒风险因子的乘性累积，最终口腔癌得分高达1029.6，远超第二名的天疱疮（81.0），得分区分度为12.7:1。

#### 失败案例：P03 可复性牙髓炎鉴别

28岁男性，深龋邻髓，冷热刺激痛但刺激移除后立即缓解，无自发痛。引擎将深龋（13.5分）排名第一，可复性牙髓炎（1.3分）排名第五。问题出在互斥抑制机制——可复性牙髓炎和不可复性牙髓炎处于同一互斥组（pulp_pathosis），由于不可复性牙髓炎也有一定基础分（来自q_pain_cold_hot_diff的分支得分），互斥抑制将可复性的得分压至0.15倍，导致深龋反超。

#### 失败案例：P15 智齿冠周炎

21岁女性，智齿区域牙龈肿痛、吞咽困难、张口受限。系统将牙龈炎（51.0分）排在第一，智齿冠周炎仅得6.0分排第五。系统缺乏针对智齿冠周炎的专属DAG问题节点，冠周炎的唯一得分来源是全局压力问题的风险因子（`pericoronitis:2`），而牙龈炎从牙周专科路径中累加了大量基础分。

### 4.5 健康对照组验证

P40（30岁女性，无任何口腔症状，口腔卫生良好）的首诊结果为TMD（12.3分），低于15分的安全阈值。该低分主要是由于选择颌面关节入口（分支4）的默认TMD基准分（3分），加上女性性别偏倚（×1.8）和压力量表（防护系数×0.5），说明引擎对无症状用户不会产生高分的误报。

## 5. 讨论

### 5.1 系统优势

测试结果表明，该系统在以下方面表现优异：

1. **黏膜病变高灵敏度：** 9例黏膜病例全部准确，尤其是癌前病变（白斑、OSF、扁平苔藓）和口腔癌的检出率100%，得益于黏膜路径的多层深挖（可刮性→形态→危险信号→持续时间→恶性红绿灯）和吸烟/槟榔风险因子的高倍乘性放大。

2. **紧急分层合理：** 红色警告仅触发于真正需要紧急处理的场景（ANUG坏死性感染、口腔外伤、槟榔相关重度纤维化、放疗后口炎），未出现在常规牙龈炎或浅龋病例中。

3. **得分动态范围大：** 从12.3（健康）到1029.6（口腔癌），约83倍的得分差，为临床优先级排序提供了清晰的信号强度。

### 5.2 系统局限

1. **牙周疾病鉴别粒度不足：** 5例牙周病例中仅3例准确。智齿冠周炎和药物性牙龈增生缺乏专属DAG问题节点，与普通牙龈炎混为一谈。牙周-牙髓联合病变、根分叉病变等复杂牙周状况同样缺乏特异性问题。

2. **互斥抑制过度激进：** 可复性/不可复性牙髓炎的互斥组（pulp_pathosis）将低分者压制至0.15倍，而临床上两者是连续谱——深龋可先引起可复性炎症，再发展为不可复性。将互斥因子从0.15调整至0.3-0.5，或引入"过渡态"中间类别，可提升鉴别准确性。

3. **放射性口炎路径缺失：** 放射性口炎（P36）是头颈部放疗后最重要的口腔并发症，但系统缺乏与之对应的DAG入口路径。患者只能从牙周路径进入，导致慢性牙周炎被排在首位。建议增加放疗史触发的专属入口。

4. **多选问题的引擎缺陷：** 如前所述，当前引擎的`answerBranch()`方法对多选问题存在原生路由缺陷——首个分支执行后自动切换到下一个全局问题，导致后续多选分支应用到错误节点。此问题在浏览器交互中被UI层的"确认按钮"临时绕过，但阻碍了自动化和批量测试。

### 5.3 测试架构反思

本测试架构具有以下推广价值：

- **零侵入性：** 通过jsdom注入导出代码，无需修改生产代码
- **高可复现性：** 确定性的DAG遍历，每次运行结果完全一致
- **低成本扩展：** 新增测试用例只需定义答案映射，无需编写额外代码
- **适用边界：** 仅适用于无外部API调用的纯客户端专家系统

局限性：模拟病例的"真实感"受限于病例设计者的临床知识；未能覆盖所有72种疾病的完整测试；缺乏与临床金标准（活检、影像）的对比验证。

## 6. 结论

本研究设计并实现了一种轻量级、零侵入的模拟患者问卷测试架构，对覆盖72种口腔疾病的DAG二叉树网络AI预问诊系统进行了30份病例的系统性测试。测试覆盖五大疾病类别、三个紧急级别和完整人口统计学维度。

**主要发现：**
1. 系统总体首诊准确率达90.0%，黏膜病变和颌面关节类疾病识别准确率达100%
2. 红色警告机制有效区分真正紧急病例（ANUG、外伤、槟榔相关OSF、放疗后口炎）与常规病例
3. 乘性风险累积机制在癌前病变/恶性肿瘤检测中产生了高达40倍的信号放大
4. 牙周类疾病鉴别粒度不足（60%准确率），主要瓶颈在于缺乏智齿冠周炎、药物性牙龈增生等亚类的专属DAG问题
5. 健康对照组首诊得分12.3分，未产生有意义的误报

**改进建议：**
1. 增加智齿冠周炎、放射性口炎、药物性牙龈增生的专属DAG入口路径
2. 将可复性/不可复性牙髓炎的互斥因子从0.15调整至0.3-0.5
3. 修复多选问题的引擎路由缺陷，支持批量分支应用
4. 扩展测试至72种疾病的完整覆盖

本研究提出的测试架构具有通用性，可推广至其他基于规则推理的医学预诊专家系统的快速验证。
